## 八、关键业务逻辑

### 8.1 今日任务生成逻辑

```typescript
// 获取今日应显示的任务
function getTodayTasks(tasks: Task[], date: Date): Task[] {
  const dayOfWeek = date.getDay(); // 0=周日, 1=周一 ...
  return tasks.filter(task => {
    if (!task.isActive) return false;
    if (task.repeatType === 'daily') return true;
    if (task.repeatType === 'once') {
      // 检查 createdAt 日期是否就是今天
      return isSameDay(task.createdAt, date);
    }
    if (task.repeatType === 'weekly') {
      return task.repeatDays.includes(dayOfWeek);
    }
    return false;
  }).sort((a, b) => a.sortOrder - b.sortOrder);
}
```

### 8.2 连续打卡计算逻辑

```typescript
function calculateStreak(checkins: CheckIn[]): { current: number, longest: number } {
  // 1. 获取所有有打卡记录的日期（去重，排序）
  const dates = [...new Set(checkins.map(c => c.date))].sort();
  
  // 2. 计算当前连续天数（从今天往前数）
  let current = 0;
  let checkDate = new Date();
  // 如果今天还没打卡，从昨天开始算
  if (!dates.includes(formatDate(checkDate))) {
    checkDate = subDays(checkDate, 1);
  }
  while (dates.includes(formatDate(checkDate))) {
    current++;
    checkDate = subDays(checkDate, 1);
  }
  
  // 3. 计算历史最长连续
  let longest = 0, streak = 1;
  for (let i = 1; i < dates.length; i++) {
    if (differenceInDays(dates[i], dates[i-1]) === 1) {
      streak++;
    } else {
      longest = Math.max(longest, streak);
      streak = 1;
    }
  }
  longest = Math.max(longest, streak);
  
  return { current, longest };
}
```

### 8.3 打卡操作逻辑

```typescript
// POST /api/checkins
async function checkIn(taskId: number, quality?: number, photo?: string) {
  const today = formatDate(new Date()); // YYYY-MM-DD
  
  // 检查今日是否已打卡
  const existing = await findCheckIn(taskId, today);
  if (existing) {
    throw new Error('今日已打卡');
  }
  
  // 创建打卡记录
  const checkin = await createCheckIn({
    taskId,
    date: today,
    completedAt: new Date().toISOString(),
    quality,
    photoUrl: photo,
  });
  
  // 更新 streak
  await updateStreak();
  
  return checkin;
}
```

### 8.4 离线支持策略

1. 前端使用 `localStorage` 缓存任务列表和最近打卡记录
2. 打卡操作先写本地，标记 `synced: false`
3. 网络恢复后自动批量同步未同步记录
4. 冲突处理：以服务端时间为准，本地为备份

## 十、数据备份设计

### 10.1 自动备份策略

| 项目 | 说明 |
|------|------|
| 备份时机 | 服务启动时立即备份一次，之后每 6 小时自动备份 |
| 备份方式 | 直接复制 SQLite 数据库文件（`prisma/homework.db` → `backups/homework_YYYY-MM-DD_HH-mm-ss.db`） |
| 保留策略 | 保留最近 30 份备份，过期自动清理 |
| 存储位置 | `backups/` 目录（本地保留，不提交 Git） |

### 10.2 备份 API

| 方法 | 路径 | 说明 |
|------|------|
| GET | `/api/backup` | 获取备份列表（文件名、大小、时间） |
| POST | `/api/backup` | 手动触发备份 |
| POST | `/api/backup/restore` | 从指定备份恢复（恢复前自动创建安全备份） |

### 10.3 恢复流程

```
用户请求恢复 → 自动备份当前数据库（安全备份）→ 复制目标备份到生产路径 → 服务重启生效
```

### 10.4 Git 备份

数据库文件 `prisma/homework.db` 已纳入 Git 管理，每次数据变更后提交即可实现远程备份：

```bash
git add -A
git commit -m "backup: 数据更新"
git push
```

- ✅ 一次性任务 + 每日重复 + 每周重复
- ✅ 一键打卡（≤2 次点击）
- ✅ 打卡成功动画反馈
- ✅ 完成率统计（日/周/月）
- ✅ 未完成任务列表
- ✅ 连续打卡天数展示（首页 StreakBadge）
- ✅ 响应式布局（桌面/平板/手机）
- ✅ 家长密码保护
- ✅ 一次性任务支持指定日期
- ✅ 数据库自动备份（每 6 小时，保留 30 份）
- ✅ 备份管理 API（列表 / 手动备份 / 恢复）
- ✅ 数据库文件纳入 Git 远程备份
